# Epics E001–E011: tickets 0001–0053

## E001: Foundational audio engine

Establish the complete foundational stack required to run a live audio patch: the core module abstraction, a graph for wiring modules together, concrete module implementations (sine oscillator and stereo audio output), a patch builder that resolves the graph into an executable plan, and a sound engine that drives the plan against real audio hardware.

Tickets: 0001, 0002, 0003, 0004, 0005, 0006

## E002: Post-E001 cleanup and hardening

A code review of the E001 deliverables identified improvements across real-time safety, API clarity, code conventions, flexibility, and documentation.

Tickets: 0009, 0010, 0011, 0012, 0013, 0014, 0015, 0016

## E003: Module lifecycle improvements

A set of related improvements to module lifecycle, identity, and connectivity that establish cleaner runtime behaviour and lay the infrastructure for live-coding hot-reload: 1. **AudioEnvironment / initialise** — decouple environment parameters (sample rate) from per-sample processing.

Tickets: 0017, 0018, 0019, 0021

## E004: Incidental fixes

Small, self-contained improvements noticed during other work that don't belong to a larger planned epic. Each ticket is independent and can be picked up or skipped in any order.

Tickets: 0022, 0023

## E005: Stable buffer pool and module destruction

Re-planning (hot-reload) previously zeroed all cable buffers because `ExecutionPlan` owned them and each build created a fresh allocation. This caused a 1-sample discontinuity per graph hop on every re-plan. For audio-rate cables the discontinuity is inaudible, but CV cables (slow-moving control signals modulating filter cutoffs, VCA gain, etc.) may produce an audible click.

Tickets: 0024, 0025, 0026, 0028

## E006: Integration test suite

Unit tests in `patches-core` and `patches-engine` validate individual functions and data structures in isolation, but the replanning lifecycle — where a control thread builds a new plan while the audio thread runs the old one — involves interactions across all three crates that unit tests cannot reach.

Tickets: 0029, 0030, 0031, 0032, 0033, 0034

## E007: Sizeable modules (indexed ports)

Audio modules today have a fixed, statically-described port layout with no way to express multiple ports sharing the same semantic name (e.g. `in/0`, `in/1`, `in/2`).

Tickets: 0035, 0036

## E008: Control-rate signalling

The audio engine currently has no way for the control thread to update live module parameters (e.g. oscillator frequency, filter cutoff) without rebuilding the entire `ExecutionPlan`.

Tickets: 0037, 0038, 0041

## E009: Audio-thread-owned module pool and live state preservation

Replace the current `ModuleInstanceRegistry` / `held_plan` state-preservation mechanism with an audio-thread-owned module pool, symmetric with the existing buffer pool. Module instances survive replans by remaining in the pool rather than being moved across the thread boundary. State preservation becomes automatic and works on every replan, not only in the channel-full retry edge case.

Tickets: 0042, 0043, 0044, 0045

## E010: Off-thread module deallocation via cleanup ring buffer

When the audio callback adopts a new `ExecutionPlan`, it currently drops tombstoned `Box<dyn Module>` values inline (see `patches-engine/src/engine.rs` — the tombstone loop inside `build_stream`). Dropping a heap-allocated value calls `dealloc`, which is not bounded in time and violates the project's no-allocation-on-audio-thread invariant.

Tickets: 0051, 0052, 0053

## E011: Sound engine internal decomposition

Following the audio-thread module pool work (E009), several internal structures in `patches-engine` were left with mixed responsibilities and vtable dispatch in performance-critical paths.

Tickets: 0047, 0048, 0049, 0050
